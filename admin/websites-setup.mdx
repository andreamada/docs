## Resource Allocation

Configure disk space and bandwidth limits for newly created websites.

### Disk Space Configuration

<ParamField path="disk" type="number" default={100} required>
  **Disk Space per Website (MB)**
  
  Default storage allocated to each new website.
  
  - **Minimum:** 50 MB
  - **Maximum:** 99,999 MB (~97 GB)
  - **Recommended:** 100-500 MB for small sites, 1000+ MB for media-heavy sites
</ParamField>

### Bandwidth Management

<ParamField path="bw" type="number" default={30000}>
  **Monthly Bandwidth (MB)**
  
  Maximum data transfer allowed per website per month.
  
  - **Minimum:** 0 (unlimited)
  - **Maximum:** Unlimited
  - **Default:** 30,000 MB (30 GB)
  - **Note:** Set to `0` for unlimited bandwidth
</ParamField>

<ParamField path="bw_warning" type="number" default={30000}>
  **Bandwidth Warning Threshold (MB)**
  
  Send warning notifications when bandwidth usage exceeds this limit.
  
  - **Minimum:** 0
  - **Purpose:** Proactive monitoring to prevent service interruptions
  - **Recommendation:** Set to 80-90% of total bandwidth limit
</ParamField>

<Tip>
**Resource Planning Tips:**
- Monitor average resource usage before adjusting limits
- Consider your server capacity when setting defaults
- Set bandwidth warnings below the hard limit to give users time to upgrade
- Allocate more disk space for e-commerce or media-heavy websites
</Tip>

## User Limitations

Set limits and restrictions on website creation to prevent abuse and manage system resources.

### Website Quantity Limits

<ParamField path="quantity_website" type="number" default={5} required>
  **Max Websites per User**
  
  Maximum number of websites each user can create.
  
  - **Minimum:** 1
  - **Maximum:** 99,999
  - **Default:** 5
  - **Use Case:** Prevent resource hogging and encourage premium upgrades
</ParamField>

### Creation Delay Protection

<ParamField path="delay_create" type="number" default={60} required>
  **Creation Delay (seconds)**
  
  Minimum time required between consecutive website creations.
  
  - **Minimum:** 1 second
  - **Maximum:** 99,999 seconds
  - **Default:** 60 seconds (1 minute)
  - **Purpose:** Prevent automated abuse and reduce server load
</ParamField>

### Balance Monitoring

<ParamField path="threshold" type="number" default={50} required>
  **Balance Warning Threshold**
  
  Notify users when their account balance drops below this amount.
  
  - **Minimum:** 1
  - **Maximum:** 99,999
  - **Default:** 50
  - **Purpose:** Ensure users maintain sufficient balance for services
</ParamField>

<Warning>
**Anti-Abuse Considerations:**
- Set reasonable website limits to prevent resource exhaustion
- Use creation delays to prevent automated bot attacks
- Monitor user behavior patterns for unusual activity
- Consider implementing tiered limits based on user subscription levels
</Warning>

## Trial & Expiry Settings

Configure trial periods for new websites and manage expiration policies.

### Trial Period Configuration

<ParamField path="expired" type="number" default={3} required>
  **Trial Period (days)**
  
  Number of days for free trial websites before requiring payment.
  
  - **Minimum:** 1 day
  - **Maximum:** 99,999 days
  - **Default:** 3 days
  - **Common Values:** 3, 7, 14, or 30 days
</ParamField>

<ParamField path="expired_delete_delay" type="number" default={3} required>
  **Deletion Grace Period (days)**
  
  Days before expired websites are permanently deleted.
  
  - **Minimum:** 1 day
  - **Maximum:** 99,999 days
  - **Default:** 3 days
  - **Purpose:** Give users time to renew before data loss
</ParamField>

### Expiry Notifications

<ParamField path="expiry_date_days" type="number" default={15} required>
  **Expiry Warning Period (days)**
  
  Send expiration warnings this many days before website expiry.
  
  - **Minimum:** 1 day
  - **Maximum:** 90 days
  - **Default:** 15 days
  - **Related:** [Website Expiry Management](/admin/WebsiteExpire)
</ParamField>

<Steps>
  <Step title="Trial Period Begins">
    User creates a new website and trial period starts automatically
  </Step>
  <Step title="Expiry Warning Sent">
    System sends notification X days before expiration (configurable)
  </Step>
  <Step title="Trial Expires">
    Website expires after trial period if not renewed
  </Step>
  <Step title="Grace Period">
    Website enters grace period before permanent deletion
  </Step>
  <Step title="Permanent Deletion">
    Website and all data are permanently deleted after grace period
  </Step>
</Steps>

<Note>
**Trial Period Best Practices:**
- Longer trials (7-14 days) increase conversion rates
- Send multiple reminders before expiration
- Provide easy renewal options in notification emails
- Consider offering trial extensions for engaged users
</Note>

## Helpdesk Ticket Automation

Configure automatic ticket cleanup and closure to maintain a clean helpdesk system.

### Ticket Retention Management

<ParamField path="retention_days" type="number" default={365}>
  **Ticket Retention Period (days)**
  
  Automatically delete closed tickets after this many days.
  
  - **Minimum:** 0 (never delete)
  - **Maximum:** 3,650 days (10 years)
  - **Default:** 365 days (1 year)
  - **Purpose:** Archive old tickets and reduce database size
  - **Note:** Set to `0` to disable automatic deletion
</ParamField>

### Auto-Close Configuration

<ParamField path="autoclose_awaiting_days" type="number" default={14}>
  **Auto-Close Awaiting Client Tickets (days)**
  
  Automatically close tickets in "Awaiting Client" status after this many days.
  
  - **Minimum:** 0 (never auto-close)
  - **Maximum:** 365 days
  - **Default:** 14 days
  - **Purpose:** Close inactive tickets waiting for client response
  - **Note:** Set to `0` to disable auto-close
</ParamField>

<Warning>
**Automation Notice:** Set values to `0` to disable automation for either retention or auto-close features.
</Warning>

### Automation Workflow

<Steps>
  <Step title="Ticket Created">
    Support ticket is created by user or system
  </Step>
  <Step title="Status: Awaiting Client">
    Ticket status changes to "Awaiting Client" after support response
  </Step>
  <Step title="Auto-Close Timer Starts">
    Timer begins counting days since last client response
  </Step>
  <Step title="Auto-Close Triggered">
    Ticket automatically closes after configured days (if enabled)
  </Step>
  <Step title="Retention Period">
    Closed ticket remains in system for retention period
  </Step>
  <Step title="Permanent Deletion">
    Ticket is permanently deleted after retention period expires (if enabled)
  </Step>
</Steps>

<Tip>
**Helpdesk Automation Best Practices:**
- **Retention Period:** Keep tickets for at least 1 year for reference
- **Auto-Close:** 7-14 days is typical for awaiting client tickets
- **Notifications:** Send reminders before auto-closing tickets
- **Exceptions:** Consider manual review for high-priority tickets
- **Compliance:** Check data retention requirements for your industry
</Tip>

## Troubleshooting

<AccordionGroup>
  <Accordion title="Websites Not Being Created" icon="circle-exclamation">
    **Common Issues:**
    - Disk space limit too low for website requirements
    - User has reached maximum website limit
    - Creation delay timer still active
    - Insufficient server resources
    
    **Solutions:**
    - Increase disk space allocation
    - Raise or remove website quantity limit
    - Check creation delay settings
    - Monitor server capacity and upgrade if needed
  </Accordion>
  
  <Accordion title="Trial Periods Not Working" icon="calendar-xmark">
    **Common Issues:**
    - Trial period set to 0 or very low value
    - Expiry automation not running
    - Cron jobs not configured
    - Date calculation errors
    
    **Solutions:**
    - Verify trial period is set to reasonable value (3-30 days)
    - Check cron job configuration
    - Review expiry automation logs
    - Test with a sample website
  </Accordion>
  
  <Accordion title="Helpdesk Automation Not Running" icon="ticket">
    **Common Issues:**
    - Automation values set to 0 (disabled)
    - Cron jobs not executing
    - Database connection issues
    - Ticket status not matching criteria
    
    **Solutions:**
    - Verify automation values are greater than 0
    - Check cron job logs
    - Test database connectivity
    - Review ticket status workflow
  </Accordion>
</AccordionGroup>
