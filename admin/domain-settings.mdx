## Dynadot API Settings

Configure your Dynadot API token to enable automated domain registration and management.

### API Token Configuration

<ParamField path="api_token" type="string" required>
  **API Token**
  
  Your Dynadot API access token for domain registration.
  
  - **Source:** [Dynadot API Settings](https://www.dynadot.com/account/domain/setting/api.html)
  - **Format:** Alphanumeric string
  - **Security:** Keep this token confidential
  - **Purpose:** Enables automated domain registration via API
</ParamField>

### Getting Your Dynadot API Token

<Steps>
  <Step title="Log in to Dynadot">
    Visit [Dynadot.com](https://www.dynadot.com) and log in to your account
  </Step>
  <Step title="Navigate to API Settings">
    Go to **Account → Domain → Settings → API** or visit [Dynadot API Settings](https://www.dynadot.com/account/domain/setting/api.html)
  </Step>
  <Step title="Enable API Access">
    If not already enabled, activate API access for your account
  </Step>
  <Step title="Generate API Key">
    Create or copy your existing API key
  </Step>
  <Step title="Configure in Admin Panel">
    Paste the API token into the Domain API Settings page and save
  </Step>
</Steps>

<Info>
**API Requirements:** You need a valid Dynadot account with API access enabled to use domain registration features.
</Info>

<Warning>
**Security Best Practices:**
- Never share your API token publicly
- Store the token securely using environment variables
- Regenerate the token if compromised
- Restrict API access to trusted IP addresses in Dynadot settings
- Monitor API usage for unusual activity
</Warning>

## Default Nameservers

Configure default nameservers that will be automatically assigned to newly registered domains.

### Nameserver Configuration

<ParamField path="ns_1" type="string" required>
  **Primary Nameserver (NS1)**
  
  Default primary nameserver for new domains.
  
  - **Format:** `ns1.example.com` or IP address
  - **Purpose:** Primary DNS resolution
  - **Example:** `ns1.yourdomain.com`
</ParamField>

<ParamField path="ns_2" type="string" required>
  **Secondary Nameserver (NS2)**
  
  Default secondary nameserver for new domains.
  
  - **Format:** `ns2.example.com` or IP address
  - **Purpose:** Backup DNS resolution and redundancy
  - **Example:** `ns2.yourdomain.com`
</ParamField>

### Nameserver Best Practices

<Accordion title="Why Two Nameservers?" icon="server">
**Redundancy and Reliability:**
- **Primary (NS1):** Main DNS server for domain resolution
- **Secondary (NS2):** Backup server if primary fails
- **Requirement:** Most registrars require at least 2 nameservers
- **Best Practice:** Use geographically distributed servers
- **Uptime:** Ensures domain remains accessible if one server fails
</Accordion>

<Accordion title="Common Nameserver Providers" icon="cloud">
**Popular DNS Providers:**

| Provider | NS1 | NS2 |
|----------|-----|-----|
| **Cloudflare** | `ns1.cloudflare.com` | `ns2.cloudflare.com` |
| **AWS Route 53** | `ns-xxx.awsdns-xx.com` | `ns-xxx.awsdns-xx.net` |
| **Google Cloud DNS** | `ns-cloud-a1.googledomains.com` | `ns-cloud-a2.googledomains.com` |
| **DigitalOcean** | `ns1.digitalocean.com` | `ns2.digitalocean.com` |
| **Custom** | `ns1.yourdomain.com` | `ns2.yourdomain.com` |
</Accordion>

<Tip>
**Nameserver Setup Tips:**
- Use reliable, high-performance DNS providers
- Consider using Cloudflare for free CDN and DDoS protection
- Ensure nameservers are properly configured before assigning to domains
- Test DNS propagation after configuration
- Keep nameserver records consistent across all domains
</Tip>

## Default CNAME Record

Configure the default CNAME record that websites will point to.

<ParamField path="record_domain" type="string" required>
  **Default CNAME Record**
  
  Default CNAME pointing domain for websites.
  
  - **Format:** `yourdomain.com` (without protocol)
  - **Purpose:** Points customer domains to your hosting infrastructure
  - **Default:** Uses `PRIMARY_DOMAIN` variable
  - **Example:** `hosting.yourdomain.com`
</ParamField>

### Understanding CNAME Records

<Accordion title="What is a CNAME Record?" icon="link">
**CNAME (Canonical Name) Record:**

A CNAME record creates an alias from one domain name to another. In this context:

- **Customer Domain:** `customer-site.com`
- **CNAME Points To:** `hosting.yourdomain.com`
- **Result:** Traffic to `customer-site.com` is routed to your hosting server

**Example Configuration:**
```
customer-site.com  →  CNAME  →  hosting.yourdomain.com
```

**Benefits:**
- Centralized hosting management
- Easy server migration (change one record instead of many)
- Load balancing and traffic distribution
- SSL certificate management
</Accordion>

<Accordion title="CNAME vs A Record" icon="circle-question">
**Key Differences:**

| Feature | CNAME | A Record |
|---------|-------|----------|
| **Points To** | Domain name | IP address |
| **Flexibility** | Easy to update | Requires IP changes |
| **Use Case** | Aliasing domains | Direct IP mapping |
| **Root Domain** | Not allowed | Allowed |
| **Subdomain** | Allowed | Allowed |

**When to Use CNAME:**
- Pointing multiple domains to one hosting server
- Need flexibility to change underlying IP
- Using CDN or load balancer

**When to Use A Record:**
- Root domain configuration
- Direct IP mapping required
- Maximum DNS query speed
</Accordion>

## Top Level Domain (TLD) Configuration

Define which TLDs (domain extensions) are available for domain availability checking.

### Primary TLD List

<ParamField path="domain_check_default" type="textarea" required>
  **Primary TLD List**
  
  List of primary TLDs to check for domain availability.
  
  - **Format:** One TLD per line, without the dot
  - **Example:** `com`, `net`, `org`
  - **Purpose:** Default TLDs shown in domain search
  - **Recommendation:** Include most popular TLDs
</ParamField>

### Extended TLD List

<ParamField path="domain_check_more" type="textarea">
  **Additional TLD List**
  
  Additional TLDs for extended domain search.
  
  - **Format:** One TLD per line, without the dot
  - **Example:** `io`, `ai`, `dev`
  - **Purpose:** Extended search options for users
  - **Use Case:** Niche or country-specific TLDs
</ParamField>

<Warning>
**TLD Format Requirements:** Enter TLDs without the dot, one per line. 

**Correct:**
```
com
net
org
```

**Incorrect:**
```
.com, .net, .org
```
</Warning>

## Best Practices

<Tip>
**Domain API Configuration Tips:**
- **Test API Token:** Verify token works before going live
- **Use Reliable DNS:** Choose established nameserver providers
- **Monitor API Limits:** Track Dynadot API usage and quotas
- **Update TLD Lists:** Regularly review and update available TLDs
- **Document Changes:** Keep records of configuration modifications
- **Backup Settings:** Export configuration before major changes
</Tip>

<Warning>
**Critical Considerations:**
- **API Token Security:** Never expose token in client-side code
- **DNS Propagation:** Allow 24-48 hours for nameserver changes
- **TLD Availability:** Not all TLDs support all features
- **Cost Management:** Monitor domain registration costs
- **Compliance:** Ensure TLDs comply with local regulations
</Warning>

## Troubleshooting

<AccordionGroup>
  <Accordion title="API Token Not Working" icon="key">
    **Common Issues:**
    - Invalid or expired API token
    - API access not enabled in Dynadot account
    - IP address not whitelisted
    - Account suspended or restricted
    
    **Solutions:**
    - Verify token is correctly copied (no extra spaces)
    - Check API access is enabled in Dynadot settings
    - Whitelist your server IP in Dynadot
    - Contact Dynadot support for account issues
    - Regenerate API token if necessary
  </Accordion>
  
  <Accordion title="Domain Registration Failing" icon="circle-xmark">
    **Common Issues:**
    - Insufficient account balance
    - Domain already registered
    - TLD not supported by Dynadot
    - Invalid domain name format
    - API rate limits exceeded
    
    **Solutions:**
    - Check Dynadot account balance
    - Verify domain availability first
    - Confirm TLD is supported by Dynadot
    - Validate domain name format
    - Implement rate limiting in your application
  </Accordion>
  
  <Accordion title="Nameserver Issues" icon="server">
    **Common Issues:**
    - Nameservers not resolving
    - DNS propagation delays
    - Incorrect nameserver format
    - Nameserver not responding
    
    **Solutions:**
    - Test nameservers with `nslookup` or `dig`
    - Wait 24-48 hours for DNS propagation
    - Verify nameserver format (no http://, no trailing slash)
    - Check nameserver provider status
    - Use multiple DNS checkers to verify propagation
  </Accordion>
  
  <Accordion title="CNAME Record Problems" icon="link">
    **Common Issues:**
    - CNAME not resolving
    - Incorrect CNAME format
    - CNAME conflicts with A record
    - SSL certificate issues
    
    **Solutions:**
    - Verify CNAME format (domain only, no protocol)
    - Remove conflicting A records
    - Ensure CNAME points to valid domain
    - Configure SSL for CNAME target
    - Test with DNS lookup tools
  </Accordion>
  
  <Accordion title="TLD List Not Updating" icon="list">
    **Common Issues:**
    - Cache not cleared
    - Form not saved properly
    - Invalid TLD format
    - Database update failed
    
    **Solutions:**
    - Clear browser and server cache
    - Verify save button was clicked
    - Check TLD format (no dots, one per line)
    - Review server logs for errors
    - Test with a simple TLD list first
  </Accordion>
</AccordionGroup>

## Security Considerations

<Warning>
**Critical Security Notes:**
- **API Token Protection:** Store securely, never expose in frontend
- **Access Control:** Restrict API settings to admin users only
- **Audit Logging:** Log all configuration changes
- **Rate Limiting:** Implement API call limits
- **Monitoring:** Track unusual API activity
- **Backup:** Regular backups of configuration
</Warning>

<Tip>
**Security Best Practices:**
- Use environment variables for API tokens
- Enable two-factor authentication on Dynadot account
- Regularly rotate API tokens
- Monitor Dynadot account for unauthorized access
- Implement IP whitelisting in Dynadot settings
- Review API usage logs periodically
</Tip>
