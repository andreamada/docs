## What is the Workflow Builder?

The Workflow Builder is a visual no-code automation tool. You connect **trigger nodes** to **action nodes** to create automated sequences that run on a schedule or in response to events.

### Example Workflows

- **Nightly cleanup** — Stop dev containers at midnight, free disk space, restart services
- **Deployment pipeline** — Pull latest code → run tests → deploy to staging → notify Slack
- **Alert response** — When CPU > 90% → scale down non-critical services → send email alert
- **New client onboarding** — User registers → deploy WordPress → send welcome email → create DNS record

## Building a Workflow

<Steps>
  <Step title="Create a new workflow">
    Go to **Workflow** and click **New Workflow**. Give it a name and description.
  </Step>

  <Step title="Add a trigger">
    Every workflow starts with a trigger:

    | Trigger | Description |
    |---------|-------------|
    | **Schedule** | Cron expression (e.g. `0 2 * * *` = 2am daily) |
    | **Event** | Platform event (new user, site deployed, backup complete, etc.) |
    | **Webhook** | Incoming HTTP request to a unique URL |
    | **Manual** | Run on-demand from the UI |
  </Step>

  <Step title="Add action nodes">
    Connect actions that execute when the trigger fires:

    | Category | Actions |
    |----------|---------|
    | **Server** | Run shell command, restart service, exec Docker command |
    | **Sites** | Deploy site, run WP-CLI, clear cache, take backup |
    | **Notifications** | Send email, webhook POST, Slack message |
    | **Logic** | If/Else condition, loop, delay, set variable |
    | **API** | HTTP GET/POST to external URLs |
    | **Files** | Upload to S3, delete old files, compress directory |
  </Step>

  <Step title="Configure conditions">
    Add conditional branches using the **If/Else** node to handle different outcomes based on command output, status codes, or variable values.
  </Step>

  <Step title="Test and activate">
    Use the **Test Run** button to execute the workflow in debug mode and inspect each node's output. When satisfied, toggle the workflow to **Active**.
  </Step>
</Steps>

## Workflow Nodes

### Trigger Nodes

| Node | Configuration |
|------|--------------|
| **Cron Schedule** | Cron expression, timezone |
| **Platform Event** | Event type filter (site.deployed, user.created, etc.) |
| **Incoming Webhook** | Auto-generated URL, optional auth token |
| **Manual Trigger** | No configuration required |

### Action Nodes

| Node | Inputs | Outputs |
|------|--------|---------|
| **Shell Command** | Server, command string | stdout, exit code |
| **WP-CLI** | Site, WP-CLI command | output, success/fail |
| **HTTP Request** | URL, method, headers, body | status code, response body |
| **Send Email** | To, subject, body (template variables) | success/fail |
| **Docker Exec** | Container name, command | output |
| **Create Backup** | Site or server, destination | backup ID |

### Logic Nodes

| Node | Description |
|------|-------------|
| **If/Else** | Branch based on a condition (variable comparison, exit code) |
| **Loop** | Iterate over a list of servers or sites |
| **Delay** | Wait N seconds/minutes before continuing |
| **Set Variable** | Store a value for use in later nodes |
| **Stop** | Terminate the workflow (optionally with a status) |

## Variables

Workflows support variable interpolation using `{{ variable_name }}` syntax:

```
{{ trigger.server_id }}      # Server that triggered the event
{{ trigger.site_domain }}    # Domain of the affected site
{{ node.shell.stdout }}      # Output from a previous shell command node
{{ env.PANEL_URL }}          # Panel URL from environment
{{ date.iso }}               # Current date/time in ISO format
```

## Workflow History

Every workflow execution is logged. Go to **Workflow → History** to see:

- Execution time and duration
- Status (success, failed, partial)
- Each node's output and exit code
- Error messages for failed nodes

<Tip>
Workflows that fail 3 times in a row are automatically **paused** and an alert is shown on the dashboard. Review the history to diagnose the issue before re-enabling.
</Tip>
